RAG란 무엇인가 — 두 국면의 흐름What RAG is — a flow in two phases

RAG는 성격이 다른 두 국면으로 나뉜다. A. 자료 준비는 한 번만 하는 오프라인 작업이고, B. 질문·검색·답변은 질문할 때마다 반복되는 온라인 작업이다.RAG divides into two phases of different character. A. Preparing the material is offline work done once; B. Question, retrieval, answer is online work repeated with every question.

def classify(spans):
    font, size = spans[0]["font"], round(spans[0]["size"], 1)

    if "Gotham-Black" in font and 15 <= size <= 17:
        return "goal_title"

STEP 4 · XML → chunks → LangChain Documents

XML → chunk

<subsection> 하나 = 청크 하나가 원칙이다. 본문은 검색 대상이 되고, Goal 번호·이름·쪽수·heading은 메타데이터(전거)가 되어 매 검색 결과에 따라붙는다.The principle: one <subsection> = one chunk. The body text becomes searchable; the Goal number, name, pages and heading become metadata (the provenance) and ride along with every retrieval result.

정본 XML은 드라이브에서 읽기 전용으로 불러오므로, 참가자 전원이 완전히 같은 판본에서 출발한다. 검색 대상이 되는 것은 소제목 + 본문이고, Goal 번호·이름·쪽수는 검색되지 않고 전거로만 실린다.The canonical XML is loaded read-only from the Drive, so every participant starts from exactly the same edition. What becomes searchable is subheading + body; the Goal number, name and pages are never searched — they ride only as provenance.

  • page_content소제목 + 본문 문단 — 이 전체가 벡터가 된다Subheading + body paragraphs — all of this becomes the vector
  • metadataid · goal · goal_name · pages · heading · part — 검색되지 않고 따라만 붙는다 — never searched, only carried along
소제목은 양쪽에 다 들어간다 — 앞에서는 검색을 돕고, 뒤에서는 출처를 밝힌다.The subheading goes on both sides — on the front it aids retrieval, on the back it names the source.
단, 인코더가 삼킬 수 있는 크기까지만But only as large as the encoder can swallow

인코더의 입력 한도는 256토큰이다. 넘으면 뒷부분이 잘린 채 벡터가 되어 검색에서 빠진다. 그래서 172단어(제목 포함)를 넘으면 문단 경계에서, 문단 하나가 그보다 길면 문장 경계에서 나눈다 — 문장은 자르지 않는다.The encoder's input limit is 256 tokens. Go over and the tail is truncated before it becomes a vector, dropping out of retrieval entirely. So beyond 172 words (title included) we split at a paragraph boundary, and if a single paragraph is longer than that, at a sentence boundary — we never cut a sentence.

한도 안에 들어오므로 1 소제목 = 1 청크 그대로.They fit within the limit, so one subheading stays one chunk.

LangChain Document는 출처가 적힌 색인 카드다. 앞면(page_content)엔 검색될 본문을, 뒷면(metadata)엔 출처를 적는다. 뒷면은 검색되지 않지만, 카드가 뽑힐 때 늘 함께 딸려 나온다 — 그것이 전거(典據, 출처 근거)다.A LangChain Document is an index card with its source written on it. The front (page_content) holds the text to be searched; the back (metadata) holds the source. The back is never searched, but it always comes out with the card when the card is drawn — that is provenance.

소제목만은 양면에 모두 적는다. 앞면에 두는 이유는 나뉜 조각 때문이다 — part 2/2는 본문 중간 문단들만 갖고 있어서, 소제목이 없으면 무슨 주제인지 알 수 없는 고아 문단이 된다. 제목을 붙여야 검색이 그 조각을 찾아낸다.The subheading alone is written on both faces. It goes on the front because of the split pieces — a part 2/2 holds only middle paragraphs, and without the subheading it is an orphan passage whose subject cannot be told. Attach the title and retrieval can find that piece.

STEP 5 · Load the encoder · embed every chunk

Chunk → Vector

각 청크를 384개의 숫자로 바꾼다. 이 좌표가 의미의 위치다. GPU에서 전체가 약 1초면 끝난다.Turn each chunk into 384 numbers. Those coordinates are where its meaning sits. On a GPU the whole set takes about a second.

인코더는 드라이브에서 로컬로 불러오고(허브가 봉인돼 있으므로), normalize_embeddings=True로 정규화한다. 정규화된 벡터에서는 L2거리² = 2 − 2·코사인유사도 — L2와 코사인이 순위상 동치가 되어, 화면 숫자와 설명이 일치한다.The encoder is loaded locally from the Drive (the Hub is sealed) and normalised with normalize_embeddings=True. For normalised vectors, L2 distance² = 2 − 2·cosine similarity — L2 and cosine rank identically, so the numbers on screen match the explanation.

workshop_code.md5-1 · 5-2
embeddings = HuggingFaceEmbeddings(
    model_name=ENCODER_DIR,
    model_kwargs={"device": "cuda"},
    encode_kwargs={"normalize_embeddings": True},
)
# 각 문장 → 384개의 숫자each passage → 384 numbers
vectors = embeddings.embed_documents([c["text"] for c in chunks])
의미를 좌표로 — 384차원의 자리Meaning as coordinates — a place in 384 dimensions

글의 뜻을 384개 축 위의 한 지점으로 옮긴다. 비슷한 뜻은 가까이, 다른 뜻은 멀리 놓인다. 주제별로 자료의 자리를 정해 두는 분류 체계와 같되, 그 자리를 사람이 아니라 의미의 근접성이 결정한다.The sense of a passage is moved to a single point on 384 axes. Similar meanings land near each other, different ones far apart. It is like a classification scheme assigning each item a place by subject — except the place is decided not by a person but by proximity of meaning.

STEP 6 · Build the vector store (FAISS)

Vector DB(FAISS) 구축Building the Vector DB (FAISS)

STEP 5의 벡터가 그대로 들어간다 — 아무것도 다시 계산하지 않는다. 벡터와 원문은 따로 저장되고, 색인 번호가 둘을 잇는다.The vectors from STEP 5 go straight in — nothing is recomputed. Vectors and source text are stored separately, joined by an index number.

여기서 A(자료 준비)가 끝나고 B(질문·검색)가 시작된다. 앞의 흐름도에서 청록으로 강조된 경첩이 바로 이 색인집이다. FAISS는 벡터, 원문, id 테이블 세 저장소를 나란히 두고, 색인 번호로 즉시 맞물린다.Here A (preparing the material) ends and B (question and retrieval) begins. The hinge picked out in teal in the earlier diagram is exactly this index. FAISS keeps three stores side by side — vectors, source text, and an id table — meshed instantly by index number.

workshop_code.md6-1
store = FAISS.from_embeddings(
    text_embeddings=list(zip(
        [c["text"] for c in chunks], vectors)),
    embedding=embeddings,
    metadatas=[c["metadata"] for c in chunks],
    ids=[c["metadata"]["id"] for c in chunks],
)
# 벡터 저장소 · id 테이블 · 원문 저장소가 색인 번호로 맞물린다vector store · id table · document store mesh by index number
색인의 완성 — A에서 B로 넘어가는 자리The index complete — where A hands over to B

수천 항목을 좌표로 정리해 둔 색인. 이제 어떤 질문이 와도 전체 원문을 처음부터 읽지 않고, 색인만 뒤져 가장 가까운 항목으로 곧장 간다. 메타데이터(전거)가 함께 저장되어, 찾은 항목마다 출처가 딸려 나온다.An index arranging thousands of entries by coordinate. Whatever question arrives, we no longer read the whole text from the start — we search the index alone and go straight to the nearest entries. The metadata (the provenance) is stored alongside, so every entry found brings its source with it.

07
STEP 7 · A question becomes a vector too

Query → Vector

청크를 임베딩한 바로 그 인코더가 질문을 임베딩한다. 유일한 차이 — 질문에는 메타데이터가 없다.The question is embedded by the very same encoder that embedded the chunks. The only difference — a question has no metadata.

질문과 청크가 같은 좌표계에 놓여야 거리를 잴 수 있다. 다른 자로 잰 길이는 비교할 수 없듯, 같은 인코더로 임베딩해야 한다. 여기서 B 국면(온라인)이 실제로 돌기 시작한다.Question and chunks must sit in the same coordinate system before a distance can be measured. Just as lengths taken with different rulers cannot be compared, both must be embedded by the same encoder. This is where phase B (online) actually starts running.

workshop_code.md7-1
query = "How many people still live in extreme poverty?"

# 청크와 같은 인코더 · 같은 384차원 좌표계same encoder as the chunks · same 384-dim coordinate system
qv = embeddings.embed_query(query)

print(f"vector: {len(qv)} dims")   # → 384
같은 척도로 재기Measuring on the same scale

질문과 자료를 비교하려면 같은 척도가 필요하다. 청크를 잰 그 인코더로 질문도 재야, 둘 사이의 거리가 의미를 갖는다. 서로 다른 도량형으로 잰 값은 나란히 둘 수 없다.Comparing a question with the material requires one scale. Only if the question is measured by the encoder that measured the chunks does the distance between them mean anything. Values taken in different systems of measure cannot be set side by side.

08
STEP 8 · Distance, by hand

Distance 계산Distance, by hand

검색이란 거리를 재서 정렬하는 것, 그 이상도 이하도 아니다. FAISS 없이 넘파이로 직접 해보며, FAISS가 대신 해주는 일을 눈으로 본다.Retrieval is measuring distances and sorting them — no more, no less. We do it ourselves in NumPy, without FAISS, and see with our own eyes what FAISS does on our behalf.

쿼리 벡터와 모든 청크 벡터의 차이를 제곱해 더하고 제곱근을 취한다 — 그게 L2 거리다. 정렬하면 가장 가까운 청크가 위로 온다. 다음 단계에서 FAISS 결과와 완전히 같음을 확인한다.Take the difference between the query vector and every chunk vector, square it, sum, and take the square root — that is L2 distance. Sort, and the nearest chunk rises to the top. In the next step we confirm this is exactly identical to what FAISS returns.

workshop_code.md8-1
V = store.index.reconstruct_n(0, store.index.ntotal)  # 모든 청크every chunk
Q = np.array(qv)                                    # 질문 하나one question

distances = []
for v in V:
    diff = v - Q                        # 384개의 차이384 differences
    dist = np.sqrt((diff ** 2).sum())   # 제곱→합→제곱근square→sum→root
    distances.append(dist)

order = np.argsort(distances)             # 가까운 순 정렬sort nearest first
블랙박스를 열어 본다Opening the black box

검색이 마법처럼 보이지만, 실은 손으로 잴 수 있는 산술이다. 원리를 직접 계산해 보는 것은, 도구가 내놓은 결과를 그대로 믿기 전에 그 결과가 어떻게 나왔는지 스스로 확인하는 태도다 — 인문학 연구가 근거를 손수 대조해 온 방식 그대로.Retrieval looks like magic, but it is arithmetic you can do by hand. Working the principle out yourself is the habit of checking how a result was arrived at before trusting what a tool hands you — exactly how humanities scholarship has always collated its evidence.

STEP 9 · FAISS retrieval + provenance

FAISS 검색 + 전거FAISS retrieval + provenance

STEP 8에서 손으로 한 것을 FAISS가 한 줄로 한다. 결과가 같음을 확인한 뒤, 각 결과에 출처(Goal·subsection·쪽·거리)를 붙인다.What we did by hand in STEP 8, FAISS does in one line. Having confirmed the results match, we attach to each one its source (Goal · subsection · pages · distance).

"방금 손으로 한 게 이거였다." 손계산과 FAISS의 상위 3개가 정확히 일치함을 보여주고, 메타데이터로 전거를 표시한다 — 메타데이터는 검색되지 않지만 결과에 늘 딸려 나온다.“That is what we just did by hand.” We show the top 3 from the hand computation and from FAISS agreeing exactly, then display the provenance from the metadata — which is never searched, yet always comes out with the result.

workshop_code.md9-1 · 9-2
results = store.similarity_search_with_score(query, k=3)
print("STEP 8 (by hand):", hand_top3)
print("STEP 9 (FAISS) :", faiss_top3)
print("match          :", hand_top3 == faiss_top3)   # True

# 각 결과에 전거를 붙인다attach the provenance to each result
print(f"{m['id']} · Goal {m['goal']} {m['goal_name']} · pp. {m['pages']}")
전거(典據)의 부착 — 출처 없는 결과는 못 믿는다Attaching provenance — a result without a source cannot be trusted

찾은 대목마다 "어느 목표, 어느 절, 몇 쪽"을 명기한다. 인용에 반드시 출처를 다는 인문학의 규율 그대로다. 근거를 댈 수 없는 검색 결과는 신뢰할 수 없다.For every passage found, state which Goal, which section, which pages. It is the humanities' own discipline of always citing a source. A retrieval result whose grounds cannot be given is not trustworthy.

STEP 10 · Generation with provenance (Qwen3-0.6B)

생성 + 전거Generation + provenance

검색기가 근거를 찾고, LLM이 그것을 답으로 엮는다. 모든 답변은 그 답이 딛고 선 출처와 함께 제시된다.The retriever finds the grounds; the LLM weaves them into an answer. Every answer is presented together with the sources it stands on.

Qwen3는 causal 모델(AutoModelForCausalLM)이자 chat 모델이라, 입력을 chat 템플릿으로 감싸고 답변만 잘라낸다. 핵심은 before/after 대비다.Qwen3 is a causal model (AutoModelForCausalLM) and a chat model, so we wrap the input in a chat template and slice out just the answer. The point is the before/after contrast.

BEFORE · context 없음BEFORE · no context

기억만으로 답한다Answering from memory alone

검색도 근거도 없이 모델의 기억에서 답을 꺼낸다. 그럴듯하지만 낡았을 수 있고, 틀렸을 수 있고, 확인할 출처가 없다.The answer is drawn from the model's memory, with no retrieval and no grounds. It is plausible, but it may be stale, it may be wrong, and there is no source to check.

answer = llm.invoke(query) # 출처 없음. 검증 불가.
answer = llm.invoke(query) # No source. Cannot be verified.
AFTER · 검색된 근거 위에서AFTER · on retrieved grounds

보고서에 딛고 선 답An answer standing on the report

근거를 검색해 LLM에 넣고, 답변을 출처와 함께 보여준다. 이제 모든 진술이 열어 확인할 수 있는 named source 위에 선다.The grounds are retrieved, fed to the LLM, and the answer is shown together with its sources. Now every statement stands on a named source you can open and check.

ask_with_sources(query) # 답변 + Sources: [G01-S01] # Goal 1 No poverty · pp. 10-11
ask_with_sources(query) # answer + Sources: [G01-S01] # Goal 1 No poverty · pp. 10-11
workshop_code.md10-3
def ask_with_sources(q, k=3):
    results = store.similarity_search_with_score(q, k=k)   # 1. 검색retrieve
    context = "\n\n".join(doc.page_content for doc, _ in results)
    prompt = f"Use only the following sources... {context}\n\nQuestion: {q}"
    answer = llm.invoke(prompt)                            # 3. 생성generate
    print(answer)
    print("\nSources")                                    # 4. 전거provenance
    for doc, _ in results:
        m = doc.metadata
        print(f"  [{m['id']}] Goal {m['goal']} · pp. {m['pages']}")
같은 질문, 두 개의 답. 첫째는 확신에 차 있으나 확인할 수 없고, 둘째는 열어볼 수 있는 출처 위에 선다. 그 대비가 전거 우선 RAG의 전부다.One question, two answers. The first is confident but uncheckable; the second stands on sources you can open. That contrast is the whole of provenance-first RAG.

오늘 두 번의 실패를 봤다. STEP 2는 구조를 모른 채 글자 수로 잘라서 실패했고, STEP 3의 옛 방식은 구조를 추측하려다 실패했다. 두 실패의 원인은 같다 — 자료의 구조를 먼저 확정하지 않았다는 것.We saw two failures today. STEP 2 failed by cutting on character count without knowing the structure; the old approach in STEP 3 failed by trying to guess the structure. Both failures have the same cause — the structure of the material was not established first.

구조를 확정하고 나면 나머지는 따라온다. 무엇을 버릴지가 분명해지고, 어디서 자를지가 분명해지고, 잘린 조각마다 제 주소가 붙는다. 전거는 마지막에 덧붙이는 장식이 아니라, 구조를 지킨 결과로 저절로 남는 것이다.Once the structure is settled, the rest follows. What to discard becomes clear, where to cut becomes clear, and every piece cut carries its own address. Provenance is not an ornament added at the end — it is what remains of its own accord when the structure has been kept.

그러므로 RAG에서 가장 먼저 할 일은 임베딩 모델을 고르는 것이 아니라, 내 자료가 어떻게 생겼는지 아는 것이다.So the first thing to do in RAG is not to choose an embedding model, but to know what your own material looks like.

이 워크샵이 남기는 것은 챗봇 한 개가 아니라, 여러분의 자료에 그대로 옮겨 쓸 수 있는 방법이다. 실록이든 관보든 문학 텍스트든 철학 원전이든 — 자료마다 구조가 놓인 자리는 다르지만, 구조를 먼저 읽고 거기에 정제를 맞춘다는 순서는 같다. 그 순서를 세우는 자리에서 오늘의 작업이 다시 쓰인다.What this workshop leaves you is not one chatbot but a method you can carry straight over to your own material. Veritable records, official gazettes, literary texts, philosophical originals — where the structure sits differs with every source, but the order is the same: read the structure first, then fit the cleaning to it. Wherever you set that order, today's work is written again.
